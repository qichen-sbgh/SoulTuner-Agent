## 1. 你感觉"粗糙"的三个根因（诊断）

### 1.1 检索控制流：两套决策层互相打架

`agent/music_graph.py:analyze_intent` 让 LLM 输出 `intent_type` + `use_graph/use_vector`。
但 `retrieval/hybrid_retrieval.py:retrieve()` 又叠了至少 5 段关键词兜底**二次改判**：
- 实体查询清理未显式的 genre/mood/scenario hint（211-235）
- 纯音乐关键词 → 强制 `language=Instrumental` + graph-only（237-248）
- 情绪词 + 标签 + 无实体 → 把 graph_search 升级成 hybrid（250-272）
- 复合词（"国摇"）→ 推断 language/region（274-286）

每一段单看都合理，但叠在一起：**"到底跑哪个引擎、用哪些过滤"散落在 LLM + 5 处正则里，谁也说不清最终态**。这就是"混乱"。而且它们能互相覆盖（LLM 判 hybrid，兜底又改回 graph-only）。

> 根本原因：**用"意图类型"去硬门控"引擎开关"**。一旦 query 是"七分情绪三分有个歌手名"，硬门控就必然要打补丁。补丁越多越乱。

### 1.2 硬约束 vs 软排序没分离

当前 mood/scenario/genre 被当作**图谱过滤器**（WHERE 条件）。问题：
- 软维度（情绪/场景/氛围）做硬过滤 → 要么过窄（AND 到空集），要么靠兜底放宽。
- 这是 faceted search 的经典坑：**主观维度不该进 WHERE，该进 ORDER BY（排序）**。

### 1.3 语言耦合

- few-shot 示例全中文、`graph_mood_filter` 取值全中文（开心/悲伤…）、`SCENARIO_TAG_MAP` 中文键。
- 英语 query → LLM 得"英→中"翻译再套词表，比中文多一跳错误；实体匹配 `Artist.name` 若存中文名，英文名要靠别名表。
- 唯一对英语友好的是 HyDE→M2D-CLAP（本就英文）。所以**英语下"语义检索还行，意图标签层最弱"**——正是你的体感。

## 2. 七问七答（每个都给主流解法 + 落到你的代码）

### Q1 · 语言切换 / 英语没有意图识别效果

**主流做法：把"界面语言"和"内部规范词表"解耦。**
- 内部用**语言无关的规范标签**（英文受控词表或 ID），LLM 把任意语言输入 → 规范标签；UI 再渲染成任意语言。这是 NLU 国际化的标准做法。
- few-shot 给**中英混合**示例（或语言中立示例）。
- 实体链接别用字符串等值：建**多语言别名表 + 嵌入兜底**（"Jay Chou"/"周杰伦"/"周董" → 同一 artist_id）。
- 声学路径继续强制英文 HyDE（已对）。
- 落地：`graph_*_filter` 的取值统一成英文规范集；`GENRE/MOOD/SCENARIO_TAG_MAP` 做成 {规范英文 → [多语言别名]} 的双向表。这样**英语和中文走同一条路**，也才谈得上"推广出海"。

### Q2 · 图检索 vs 语义检索"混乱"——主流到底怎么做？

**主流（现代 RecSys/RAG 共识）：不要二选一硬路由，永远多路并行召回 → 融合 → 排序。**

```
                ┌── 结构化/图谱召回（实体 + 标签）
query+plan ──┤── 稠密语义召回（HyDE → M2D-CLAP KNN）
                ├── 词法召回 BM25（标题/歌手/歌词）★你现在缺这路
                ├── 个性化召回（用户亲和图 / 收藏扩展）
                └── 冷启动/流行度召回（保新歌曝光）
                         ↓ 融合（RRF 或学习式）
                         ↓ 硬约束过滤（语言/实体/纯音乐 + 拉黑安全）  ← hard_constraints 在这里
                         ↓ 统一打分（相关 + 个性化 + 多样 + 探索）     ← soft_intent 在这里
                         ↓ MMR 重排 + 新鲜度
```

要点：
- **意图类型只调"各路召回权重 + 过滤严格度"，不做引擎 on/off。** 有实体→提高图谱权重；纯情绪→提高稠密权重；但每路都还在跑。→ **5 层关键词兜底全部删掉**，因为架构上不再需要"二次改判"。
- **融合别用"平等合并"**：你们把加权 RRF 改成了 equal merge，丢了排名信息。RRF 是稳健默认（`1/(k+rank)` 跨路相加），有反馈后再上学习式融合。
- **补一路 BM25/词法召回**：对"稻香""江南"这种近精确匹配，词法召回又快又准，能显著减少"语义检索找歌名找不准"的尴尬。
- 这套就是 candidate generation → filtering → ranking 的标准分层，把"图 vs 向量"从"模式之争"变成"两条并行通道"，混乱自然消失。

### Q3 · 一句话里既有场景词又有情绪词，怎么选歌？

**别把 mood AND scenario 当硬过滤（会过窄/空集）。两步走：**
1. **融进连续查询**：把场景+情绪合成一段 HyDE 声学描述（"driving energetic workout rhythm, uplifting and bright"），嵌入后 KNN。连续空间天然处理"合取"，连"悲伤但要适合跑步"这种张力也能平衡——这正是 soft_intent 自由文本优于固定槽的地方。
2. **作软加分而非硬过滤**：

```
score(song) = w1 · cos(emb_song, emb_hyde)              # 语义/声学相关
            + w2 · tag_overlap(song, {mood, scenario})  # 命中越多越高（软）
            + w3 · personalization(user, song)
            + w4 · diversity / novelty
            − penalties(dislike, 近期曝光, 同歌手过多)
```

mood 和 scenario 各自既进 HyDE、又进 `tag_overlap`，命中两者的歌自然排前，但命中一个的歌不会被排除。**冲突时让嵌入空间裁决，而不是 WHERE 子句。**

### Q4 · 冷门曝光 + 推用户可能喜欢 + 不要每次一样

这是 explore/exploit + 多样性 + 新鲜度三件事，要分开解：
- **个性化（exploit）**：用户亲和 + 标签 Jaccard + 用户向量（见 Q5）。
- **冷门曝光（explore）**：
  - 你已有 Thompson Sampling 探索槽 + `exploration_ratio`，框架对。关键是**奖励信号**（要 P4 反馈闭环喂进来，否则探索是盲的）。
  - **流行度去偏**：排序里减 `log(play_count)` 项，或用逆倾向加权，给长尾让位。
  - **内容嵌入天生解冷启动**：M2D-CLAP 能给零交互新歌嵌入 → 协同过滤做不到的事你能做，这是你的**结构性优势**，要强调和用足。
- **不要每次一样（新鲜度）**：
  - **近期曝光惩罚**：记录每次曝光列表，最近推过的衰减（P4 已列）。
  - **随机 top-k**：对分数做带温度的 softmax 采样，而非严格 argmax → 每次"好但不同"。
  - 跨会话去重。
- 统一：`最终 = 相关 + 个性化 − 近期曝光 − 流行度(给冷门让位) + 探索bonus`，再 MMR 列表内多样，再温度采样跨会话求新。**全部依赖 P4 反馈 + P1 尺子来调，不能拍脑袋。**

### Q5 · 长期记忆到底怎么设计？

**先分清记忆的 4 种类型（这是设计框架）：**
| 类型 | 内容 | 现状 | 建议 |
|---|---|---|---|
| 工作记忆 | 当前会话最近 N 轮 | chat_history + GSSC ✓ | 保留 |
| 情节记忆 | 具体事件（"3-01 深夜跳过了 X"） | GraphZep episodic | 降为**可选**，熔断 |
| 语义/画像记忆 | 稳定偏好（"偏忧郁民谣，厌 EDM"） | Profile Synthesizer | 增量化、定期巩固 |
| 行为记忆 | like/skip/play 日志 | Neo4j 关系 | **做实**（排序与训练的真信号） |

设计原则：
- **记忆要"衍生 + 衰减"，不是一味累积。** 旧偏好按时间衰减（你有 temporal decay）；定期把原始情节**巩固**成画像（像睡眠固化），而非每次现算。
- **查询条件化召回**：取与"本次 query 的情绪/场景"相关的记忆，而不只是"最近的"。
- **把用户表示成 M2D-CLAP 同空间里的向量**（用户 taste 质心）：① 排序时直接 `cos(user_vec, song_vec)`；② 为将来 GLIDE 式"长期偏好软提示注入"铺路。**让用户和物品在同一个空间**是关键升级。
- **坦率说**：GraphZep（异步 LLM 抽时序 KG）对"单用户音乐 App"可能**过度工程**——它正是你 P3 要熔断的延迟源。ROI 角度：一个**衰减的结构化画像（标签/歌手亲和，O(1) 读）+ 行为日志 + 小型情节向量库**，可能 80% 收益、20% 复杂度。建议把 GraphZep 设为可选增强，核心排序不依赖它。

### Q6 · 召回/粗排/精排/重排 合适吗？内部算法要优化吗？

- **漏斗范式对，但对你的体量过度工程。** 4 级漏斗是给 10^7~10^9 物品的；你 ~5000 首。**粗排（DSSM 式轻量打分）在万级以下没必要**——直接对召回全集做一次完整打分即可。建议：**召回 → 一次统一打分 → MMR/探索重排**，三级足够。少一级 = 少一处"混乱"。
- **内部算法要修的：**
  1. **三锚权重 0.45/0.30/0.25 是拍的，应从反馈学**（有 P4 后用逻辑回归/GBDT 学，或 pairwise）。
  2. **声学锚算法有缺陷**：现在算"候选集 OMAR 质心"再算每首到质心的距离——这衡量的是"在候选集里典不典型"，**不是"符不符合用户意图"**。锚点应对齐**查询/用户目标**，而非候选质心。这是真 bug 级的设计问题。
  3. **Graph Affinity 用 shortestPath** 每候选跑一次图遍历，贵且噪声大。改**预计算用户-物品亲和**或**嵌入点积**。
  4. OMAR 与 M2D 两个音频模型是否都要，结合 Q7 重审。

### Q7 · HyDE 对齐多模态语义，还有什么优化空间？

HyDE：LLM 写英文声学描述 → M2D-CLAP 文本编码 → KNN 歌曲音频向量。优化空间不小：
1. **跨模态 gap 校正**：CLAP 类模型文本/音频嵌入有已知"模态间隙"（各占锥形不同区域）。便宜有效的修法：减去**平均 gap 向量**，或对文本侧做 whitening/校准。
2. **HyDE 集成**：生成**多条**假设描述、嵌入后**取平均**——HyDE 集成已被证明稳定提点。
3. **★最高 ROI：用你自己的曲库训一个轻量对齐适配器。** 你有歌曲音频 + 能生成 caption → 拿 (caption, audio) 对训一个**线性/小 MLP 投影**（甚至冻结编码器只学投影），把文本嵌入"翻译"到你这批音频的分布上。对**你的目录**对齐立刻变紧。
4. **加一个文搜音的单点指标**：给定 caption 能否召回正确歌曲（Recall@K）——把"对齐质量"从整条链路里**隔离出来单独量**，否则你不知道是 HyDE 烂还是排序烂。
5. **直接嵌入兜底**：soft_intent 直接编码 + HyDE 取并集，模糊 query 更稳。

## 3. 目标架构（重构蓝图）

```
analyze_intent → 分层 plan {hard_constraints, soft_intent, hints, intent_type}
        │  (intent_type 只决定各路召回权重，不做 on/off)
        ▼
多路召回(并行)：图谱(实体+标签) ｜ 稠密(HyDE+soft_intent) ｜ BM25(★新增) ｜ 个性化 ｜ 冷启动
        ▼ RRF 融合（替回平等合并）
        ▼ 硬过滤：hard_constraints(语言/实体/纯音乐) + 安全(拉黑) ── 唯一的 WHERE
        ▼ 统一打分：w1·语义 + w2·标签软重叠 + w3·个性化(用户向量) + w4·多样/新鲜 − 惩罚(曝光/流行/同歌手)
        ▼ MMR 重排 + 温度采样(跨会话求新)
        ▼ FinalCut
```

相对现状的删与改：**删** 5 层关键词兜底、删粗排级、删"质心声学锚"；**改** 平等合并→RRF、固定权重→可学、用户→向量；**加** BM25 一路、gap 校正、对齐适配器。

## 4. 规划（接续 Codex 的 P0–P8，按 ROI 重排）

> 纪律不变：每步先过 `evaluate_outcomes`（dev 调、holdout 冻结）拿基线再改；改排序类必须有反馈/尺子，不拍脑袋。

- **✅ R1（地基）检索层重构**：已落地"五路并行召回→RRF→唯一硬过滤→统一打分→重排"，关键词兜底清零，retrieval 直接消费 hard/soft/hints；holdout 持平、dev +6pp。
- **✅ R1.5（健壮性）稀疏过滤 + 联网播放补偿**：缺标签不误杀、结果不足可补齐、实体与安全约束不放宽；dev 88%、holdout 80%。
- **R1.6 库存感知与联网约束闭环**：本地后过滤为空时统一触发联网；否定实体/`avoid` 对联网结果后过滤；返回库存与降级原因。
- **R2 语言规范化**：标签词表改 {规范英文→多语言别名}，few-shot 中英混合，实体别名表 + 嵌入链接。验收：英文版 outcome 用例与中文持平。
- **R3 反馈闭环 + 可学排序**（= Codex P4）：曝光/点击/skip/dislike 落表，三锚权重与各项系数改可学；修"声学锚对齐目标而非质心"。
- **R4 多模态对齐**：HyDE 集成 + gap 校正 + 自有曲库训轻量投影适配器；加"文搜音 Recall@K"单点指标。
- **R5 记忆瘦身**：GraphZep 设为可选+熔断；上线"衰减结构化画像 + 用户向量"。
- **R6 漏斗瘦身**：合并粗排+精排为一次打分（匹配 ~5k 体量）。
- 之后接 Codex 的 P5（Planner 蒸馏，须等表示冻结）、P6/P7/P8（部署/前端/合规）。

## 5. R1.5 失败三角与数据审计

### 5.1 数据规模不是一个数字

| 数据层 | 实际数量 | 结论 |
|---|---:|---|
| Docker Neo4j `Song` | 1128 | 与已处理目录基本一致 |
| `processed_audio` | 1127 个音频（1110 MP3 + 17 FLAC） | 当前在线索引的主体，不存在“这里有 5273 首但只入 1128 首” |
| 工作区全部 MP3 | 5273 | 其中 4000 首位于独立 `mtg_sample`，另有 `raw_ncm` 158、`online_acquired` 5 |
| `mtg_sample` | 4000 MP3 + 4000 metadata | 全部标为 English/Western，且不覆盖本轮失败中的目标艺人或韩语需求，不能靠一次盲目 ingest 解决失败 |

Neo4j 语言分布：English 567、Chinese 322、Japanese 92、Instrumental 85、Other 29、Mixed 13、空字符串 12、Cantonese 6、缺失 2；地区为 Korea 的歌曲只有 1 首，且没有规范 `Korean` 语言库存。语言/地区非空各 1114/1128，问题主要是**品类覆盖稀疏**，不是全库漏标。
