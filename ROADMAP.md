## 2. 北极星与设计原则

**北极星**：用户用任意语言的一句大白话，拿到"真的懂他当下意图"的歌；系统可评测、可复现、可低门槛部署演示。

**不可动摇的原则：**
1. **测结果不测特性**：推荐质量改动，一律用 `evaluate_outcomes`（dev 调、holdout 冻结）卡，绝不用意图分类准确率自证。
2. **先有尺子再改**：每个里程碑先拿改动前基线，改完对比 holdout 不掉分。
3. **表示先于规则**：硬约束=过滤(WHERE)，软意图=排序(ORDER BY)；能进连续空间的别塞固定词表。
4. **匹配体量**：你是 ~5000 首的个人/研究库，不是千万级，别套千万级的漏斗与复杂度。
5. **隐私贯穿**：密钥/口令永不入日志/代码/文档/commit；评测与训练数据先脱敏。

## 5. 贯穿全程的纪律（红线）

1. 不擅自 commit/push（除非明确要求）；保持 `tests/unit/` 全绿。
2. 评测：dev 调 / holdout 冻结；Planner `temperature=0`；报告记录 git sha + 生效模型 + 关键 config。
3. LLM judge（若引入）只看 query + 歌曲客观属性，**不看系统自己的 explanation**；上线前用人工金标准校准。
4. 隐私：密钥/口令绝不外泄；评测/训练数据脱敏。
5. 大改动前先给简短计划 + 拿基线；一次一个里程碑，做完汇报再继续。
6. 动了行为就更新本 ROADMAP + 对应专题文档。
